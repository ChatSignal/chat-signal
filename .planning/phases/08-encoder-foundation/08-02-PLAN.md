4. **Create `initEncoderOnStartup()` async function:**
   - Show the progress bar: remove 'hidden' class from `encoderProgress`
   - Define `onProgress(event)` callback:
     - Null-check `event.status` and `event.progress` (defensive, per Pitfall 4)
     - Map events to stage text (per locked decision):
       - `'initiate'` or `'download'`: "Downloading model..." + 0%
       - `'progress'`: "Downloading model..." + `event.progress`% (update fill width)
       - `'done'`: "Initializing encoder..." + 95%
       - `'ready'`: "Warming up..." + 99%
     - Update `encoderProgressFill.style.width` and `encoderProgressText.textContent`
   - Define `onError(message)` callback:
     - Add 'error' class to `encoderProgress`
     - Set `encoderProgressText.textContent = message`
     - If message contains 'unavailable' (final failure): setTimeout to add 'fade-out' class after 4 seconds, then add 'hidden' after 5 seconds, set `encoderReady = false`
   - Call `const result = await initEncoderWithRetry(onProgress, onError)`
   - If result is not null (success):
     - Set `encoderProgressFill.style.width = '100%'`
     - Set `encoderProgressText.textContent = ''`
     - Add 'fade-out' class after 500ms
     - Add 'hidden' class after 1300ms (500ms + 800ms fade)
     - Set `encoderReady = true`
     - Store backend info in chrome.storage.local: `chrome.storage.local.set({ encoderBackend: getBackendInfo() })`
     - Console.log: `[Encoder] Ready, backend: ${getBackendInfo().backend}`

5. **Call `initEncoderOnStartup()`** early in the initialization flow:
   - Find the existing initialization block (where settings load and WASM init happens) — it should be near the bottom of the file in the non-test initialization section
   - Call `initEncoderOnStartup()` AFTER settings are loaded but BEFORE the message listener starts processing (the encoder init is async and non-blocking — messages that arrive before encoder is ready will be processed by WASM keyword clustering as before)
   - IMPORTANT: Do NOT `await` the encoder init in the main init flow. Let it run in background. The encoder's `onProgress`/`onError` callbacks handle the UI. Messages arriving before encoder is ready are handled normally by WASM.

6. **Gate analysis display on encoder readiness** (per locked decision: "sidebar waits for encoder before displaying analysis results"):
   - In the `processMessages()` function (or wherever WASM analysis results are rendered to the DOM), add a guard:
     - If `encoderReady === false` AND `getEncoderState() === 'loading'`, skip rendering the analysis sections (mood, topics, clusters) — the progress bar is the only visible feedback during loading
     - If `encoderReady === false` AND `getEncoderState() === 'error'`, proceed with normal WASM rendering (fallback mode)
     - If `encoderReady === true`, proceed with normal rendering
   - IMPORTANT: Do NOT skip accumulating messages in StateManager or calling WASM — only skip the UI rendering of results while encoder is loading. This ensures the message buffer is populated when the encoder becomes ready.
   - The stats div (`#stats`) should still update message counts even during encoder loading — only the analysis sections (mood, topics, clusters, AI summary) are gated.

7. **Feed messages to encoder batch queue** (for Phase 10 consumption):
   - After WASM analysis is called in processMessages(), if `encoderReady`, call:
     ```javascript
     scheduleEncode(messages, (batch, embeddings) => {
       // Embeddings produced — store for Phase 10 cosine routing
       // For now, just log to confirm encoding works
       console.log(`[Encoder] Batch encoded: ${batch.length} messages, ${embeddings ? embeddings.length : 0} embeddings`);
     });
     ```
   - This wires the full pipeline: messages arrive → WASM analysis → encoder batching → embeddings cached. Phase 10 will replace the log with actual cosine routing.

8. **Reset encoder on session end:**
   - In the `startNewSession()` function, call `resetEncoder()` to clear the embedding cache (embeddings from a previous session are irrelevant). Do NOT re-init the pipeline — just clear the cache.
   - Actually, `resetEncoder()` as defined in Plan 01 clears everything including the pipeline. Instead, just clear the cache. Add a note in the code that a dedicated `clearCache()` export may be needed — for now, calling `resetEncoder()` and then `initEncoderOnStartup()` again would work but is expensive. The simpler approach: keep the encoder pipeline alive across sessions (it's already loaded), only messages change. Do NOT call resetEncoder on session end. The hash cache will naturally evict old entries via FIFO when it exceeds 2000.

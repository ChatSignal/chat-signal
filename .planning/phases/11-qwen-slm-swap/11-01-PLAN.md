Three changes in `llm-adapter.js`:

1. **Replace `parseSentimentResponse()` with keyword-scan regex parser (SLM-03):**
   Replace the entire function body. The new implementation:
   ```javascript
   function parseSentimentResponse(response) {
     const moodMatch = response.match(/MOOD:\s*([a-z]+)/i);
     const confMatch = response.match(/CONFIDENCE:\s*([0-9.]+)/i);
     const reasonMatch = response.match(/REASON:\s*(.+?)(?:\n|$)/i);

     // Completely unparseable: silent neutral fallback (locked decision)
     if (!moodMatch) {
       if (DEBUG) console.warn('[LLM] No MOOD keyword found, silent fallback. Response:', response);
       return { mood: 'neutral', confidence: 0.5, summary: '', emoji: MOOD_EMOJIS.neutral };
     }

     const validMoods = ['excited', 'positive', 'angry', 'negative', 'confused', 'neutral'];
     let mood = (moodMatch[1] || '').trim().toLowerCase();
     if (!validMoods.includes(mood)) mood = 'neutral';

     const confidence = confMatch ? Math.min(1, Math.max(0, parseFloat(confMatch[1]) || 0.5)) : 0.5;
     const reason = reasonMatch ? reasonMatch[1].trim() : '';

     return { mood, confidence, summary: reason, emoji: MOOD_EMOJIS[mood] || '😐' };
   }
   ```
   Key difference from old code: uses `response.match(/MOOD:\s*([a-z]+)/i)` regex (finds keyword anywhere) instead of `line.toUpperCase().startsWith('MOOD:')` (only matches at line start). This handles Qwen's conversational preamble.

2. **Add garbage-triggered fallback state tracking:**
   Add module-level variables near the top (after `let isInitialized = false;`):
   ```javascript
   let _inFallback = false;
   let _garbageCount = 0;
   const MAX_GARBAGE_BEFORE_FALLBACK = 2;
   ```

   In `analyzeSentiment()`, after calling `parseSentimentResponse()`, add garbage tracking logic:
   - After parsing, check if the result was from a silent fallback (mood === 'neutral' AND confidence === 0.5 AND summary === ''). If so, increment `_garbageCount`.
   - If `_garbageCount >= MAX_GARBAGE_BEFORE_FALLBACK`, set `_inFallback = true` and switch the engine to `createFallbackEngine()` for the rest of the session.
   - On a successful parse (mood found), reset `_garbageCount = 0`.

   Similarly in `generateAISummary` flow (via `summarizeBuckets`): add a `hasSummaryFormat()` helper that validates at least one line matches the emoji+category pattern. If zero lines match, increment garbage count and return the fallback summary instead. Export this validation so sidebar.js can check format. Actually, keep the validation inside `summarizeBuckets` itself: after getting the LLM response, validate with `hasSummaryFormat()`. If invalid, return `generateFallbackSummary(prompt)` as the summary text. Add the `hasSummaryFormat` function:
   ```javascript
   function hasSummaryFormat(text) {
     return text.split('\n').some(line => /\S.*:\s*\S/.test(line.trim()) && line.trim().length > 0);
   }
   ```

3. **Add `isInFallback()` and `retryLLM()` exports:**
   ```javascript
   function isInFallback() { return _inFallback; }

   async function retryLLM(progressCallback) {
     _inFallback = false;
     _garbageCount = 0;
     engine = null;
     isInitialized = false;
     isInitializing = false;
     await initializeLLM(progressCallback);
   }
   ```
   Add both to the export statement: `export { initializeLLM, summarizeBuckets, analyzeSentiment, computeFallbackSentiment, isLLMReady, resetLLM, isInFallback, retryLLM };`

Important: Do NOT reset `_garbageCount` on each call unconditionally — only reset on successful parse (locked decision).

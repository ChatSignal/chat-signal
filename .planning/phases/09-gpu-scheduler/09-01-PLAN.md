  <action>
Create `extension/sidebar/modules/gpu-scheduler.js` as a plain ES6 module (no DOM dependencies, no npm packages). Follow the research skeleton from 09-RESEARCH.md closely, implementing all six patterns:

**Constants (Claude's discretion picks from user-approved ranges):**
- `MAX_QUEUE_DEPTH = 8` (mid-range of locked 5-10 range)
- `GPU_TIMEOUT_MS = 30_000` (locked: 30-second timeout)
- `ENCODER_BURST_LIMIT = 4` (4:1 encoder:SLM fairness ratio)
- `AUDIT_SIZE = 20` (20-entry ring buffer for non-overlapping verification)

**Module-level state (private, not exported):**
- `_tail = Promise.resolve()` — promise-chain root
- `_queue = []` — pending task entries `{ type, priority, taskFn, resolve, reject }`
- `_activeTask = null` — currently executing task descriptor
- `_gpuAvailable = true` — flipped to false on permanent device loss
- `_consecutiveEncoderRuns = 0` — burst counter for fairness
- `_auditLog = []` — circular buffer of `{ type, startMs, endMs }` entries

**Exported functions (3 total):**

1. `scheduleGpuTask(type, priority, taskFn)` — Returns a Promise. If `_gpuAvailable` is false, immediately reject with `Error('gpu-unavailable')` and `console.log` the drop. Otherwise, wrap in a `new Promise((resolve, reject) => ...)`, create an entry object, call `_addToQueue(entry)`, then call `_drainQueue()`. Return the promise to the caller so they get the result or error.

2. `getStatus()` — Returns `{ queueDepth: _queue.length, activeTask: _activeTask ? { type: _activeTask.type } : null, auditLog: [..._auditLog], gpuAvailable: _gpuAvailable }`. This is the status API for other modules to query. All values have safe defaults (0, null, empty array, true) so callers never crash on a fresh scheduler.

3. `registerDevice(device)` — Attaches a `.then()` handler to `device.lost` (do NOT await — this promise never resolves during normal operation). When it resolves: log `[GPUScheduler] WebGPU device lost: ${info.message}` via `console.error`. If `info.reason !== 'destroyed'` (not intentional cleanup), set `_gpuAvailable = false` and dispatch `window.dispatchEvent(new CustomEvent('gpu-unavailable', { detail: { reason: info.reason, message: info.message } }))`.

**Internal functions (4 total, not exported):**

4. `_addToQueue(entry)` — If `_queue.length >= MAX_QUEUE_DEPTH`: find the oldest entry of the same type (`_queue.findIndex(t => t.type === entry.type)`). If found, splice and reject it with `Error('task-dropped')` and `console.log` the drop. If no same-type found, splice index 0 (oldest overall) and reject+log. Then `_queue.push(entry)`.

5. `_selectNext()` — Returns the index of the next task to execute, or -1 if queue is empty. Check if an SLM task exists in queue AND `_consecutiveEncoderRuns >= ENCODER_BURST_LIMIT` — if so, reset counter to 0 and return the index of the first SLM task (starvation prevention). Otherwise, find the first encoder task (priority === 1) — if found, increment `_consecutiveEncoderRuns` and return its index. Otherwise, reset counter to 0 and return 0 (next SLM task).

6. `_drainQueue()` — Chain off `_tail`: `_tail = _tail.then(async () => { ... }).catch(() => {})`. Inside the `.then()`: call `_selectNext()`, if -1 return. Splice the selected entry from `_queue`. Set `_activeTask = entry`. Record `startMs = Date.now()`. Execute with timeout: create `AbortSignal.timeout(GPU_TIMEOUT_MS)`, create a timeout promise that rejects with `DOMException('GPU task timed out after 30s', 'TimeoutError')` when signal aborts, then `Promise.race([entry.taskFn(), timeoutPromise])`. On success: `entry.resolve(result)`. On error: `console.error('[GPUScheduler] ${entry.type} task failed:', err)` then `entry.reject(err)`. In `finally`: record `endMs = Date.now()`, push `{ type: entry.type, startMs, endMs }` to `_auditLog`, shift if over `AUDIT_SIZE`, set `_activeTask = null`. After finally, if `_queue.length > 0`, call `_drainQueue()` recursively.

**CRITICAL implementation details:**
- The `.catch(() => {})` on `_tail` assignment prevents promise chain breakage (Pitfall 1 from research). The caller-facing promise returned by `scheduleGpuTask` propagates errors normally.
- Use `window.dispatchEvent(new CustomEvent(...))` for gpu-unavailable (not `chrome.runtime.sendMessage` — stays in sidebar page context).
- All console output uses `[GPUScheduler]` prefix. Only log errors and dropped tasks (minimal logging per locked decision).
- Do NOT log successful task completions to console (developer-only observability is via `getStatus()` and audit log, not console).

**Logging convention:** Follow existing project conventions from CONVENTIONS.md — `console.error` for failures, `console.log` for dropped tasks (they are expected behavior, not errors).

**encoder-adapter.js changes:**

1. Add import at top (after the existing `import { env, pipeline }` line):

2. In `initEncoder()`, after the WebGPU adapter detection block (around line 118 where `const adapter = await navigator.gpu.requestAdapter()` succeeds), register the device with the scheduler. The adapter itself is not the device — Transformers.js handles `requestDevice()` internally. However, we can register the device-loss watcher at a different point. Since encoder-adapter.js does not directly hold a `GPUDevice` reference (Transformers.js creates it internally), we need to access it indirectly. The safest approach: after `encoderPipeline` is successfully created (line 148-150 area, after the `await pipeline(...)` call), check if WebGPU backend was used and attempt to get the device from the adapter:
   ```javascript
   if (backendUsed === 'webgpu' && navigator.gpu) {
     try {
       const adapter = await navigator.gpu.requestAdapter();
       if (adapter) {
         const device = await adapter.requestDevice();
         registerDevice(device);
       }
     } catch (e) {
       console.log('[Encoder] Could not register device for loss detection:', e);
     }
   }
   ```
   Place this right after the warm-up run completes and before `encoderState = 'ready'` (around line 175-180). This registers the device-loss watcher so the scheduler can broadcast `gpu-unavailable` if the GPU is lost later.

3. Modify `encodeMessages()` to route through the GPU scheduler. The actual GPU work is the `await encoderPipeline(texts, { pooling: 'mean', normalize: true })` call on line 249. Wrap this single call (not the entire function, since cache lookup and cache storage are CPU work):
   - Replace lines 247-256 (the `if (newMessages.length > 0)` block) with:
   ```javascript
   if (newMessages.length > 0) {
     const texts = newMessages.map(m => m.text);
     const output = await scheduleGpuTask('encoder', 1, () =>
       encoderPipeline(texts, { pooling: 'mean', normalize: true })
     );
     const embeddings = output.tolist(); // number[][], shape [n, 384]

     newMessages.forEach((msg, i) => {
       embeddingCache.set(simpleHash(msg.text), embeddings[i]);
     });

     trimCache();
   }
   ```
   This wraps only the pipeline inference call through `scheduleGpuTask` with type `'encoder'` and priority `1`. Cache operations remain outside the scheduler (they are CPU-only).

4. The warm-up inference in `initEncoder()` (line 172: `await encoderPipeline(['warm up'], ...)`) should also go through the scheduler since it uses the GPU. Wrap it:
   ```javascript
   console.log('[Encoder] Running warm-up inference...');
   await scheduleGpuTask('encoder', 1, () =>
     encoderPipeline(['warm up'], { pooling: 'mean', normalize: true })
   );
   console.log(`[Encoder] Warm-up complete. Backend: ${backendUsed}`);
   ```

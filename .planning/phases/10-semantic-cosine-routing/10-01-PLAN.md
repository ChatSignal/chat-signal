  <name>Task 1: Create routing-config.js and cosine-router.js</name>
  <files>extension/sidebar/routing-config.js, extension/sidebar/cosine-router.js</files>
  <action>
Create two new ES modules in `extension/sidebar/`.

**routing-config.js** — Static config for cosine routing tuning. Export a single `ROUTING_CONFIG` object:
- `categories` array with entries for Questions, Issues/Bugs, and Requests only (General Chat is NOT in the array — it is the implicit default per Pitfall 4 in research).
- Each category has: `label` (string matching WASM bucket labels exactly), `threshold` (number, start at 0.30 per research recommendation — NOT 0.35, stream chat is noiser than literature domain), `seedPhrases` (array of 3-5 short representative phrases).
- `defaultLabel`: `'General Chat'` — used when no category exceeds threshold.
- `wasmSpeedThresholdMsPerMessage`: `200` — fallback trigger for slow WASM encoding.

Seed phrase selection (Claude's discretion):
- Questions: "what is this", "how does this work", "can someone explain", "why is this happening", "what does that mean"
- Issues/Bugs: "this is broken", "not working for me", "I found a bug", "getting an error", "something is wrong here"
- Requests: "please add this feature", "can you make it", "I would like to see", "it would be great if", "suggestion for improvement"

**cosine-router.js** — Standalone cosine routing module. No DOM access, no chrome APIs.

Imports: `encodeMessages` from `./encoder-adapter.js`, `ROUTING_CONFIG` from `./routing-config.js`.

Module-level state:
- `_prototypeVectors = null` (Map<string, number[]> — label to 384-dim normalized centroid)
- `_mode = 'keyword'` ('semantic' | 'keyword')

Exported functions:
1. `buildPrototypes()` — async. Collects all seed phrases from ROUTING_CONFIG.categories, calls `encodeMessages()` in a single batch, groups embeddings by category, computes centroid per category, L2-normalizes each centroid, stores in `_prototypeVectors`. Console.log the labels when done: `[CosineRouter] Prototypes built for Questions, Issues/Bugs, Requests`.
2. `classifyMessage(embedding)` — sync. Takes L2-normalized 384-dim array. Iterates ONLY non-General-Chat categories (from ROUTING_CONFIG.categories). For each, computes dot product (= cosine similarity for normalized vectors) against prototype vector. Tracks bestLabel/bestScore. If bestScore > category.threshold AND bestScore > previous bestScore, updates bestLabel. If no category wins (bestLabel stays null/undefined), returns `ROUTING_CONFIG.defaultLabel` ('General Chat'). Returns label string.
3. `classifyBatch(messages, embeddings)` — sync. Returns `embeddings.map(emb => classifyMessage(emb))`.
4. `isSemanticReady()` — returns `_prototypeVectors !== null && _mode === 'semantic'`.
5. `setSemanticMode()` — sets `_mode = 'semantic'`.
6. `setKeywordMode()` — sets `_mode = 'keyword'`. Does NOT clear `_prototypeVectors` (if encoder recovers, `setSemanticMode()` re-enables without rebuilding).
7. `getMode()` — returns `_mode` string.

Internal functions (not exported):
- `computeCentroid(vectors)` — averages array of 384-dim vectors, returns L2-normalized centroid.
- `l2Normalize(vec)` — in-place L2 normalization, returns same array. Guard for zero-norm.
- `dotProduct(a, b)` — sum of element-wise products.

CRITICAL: L2-normalize the centroid after averaging. The centroid of normalized vectors is NOT normalized (Pitfall 1 in research). Without this, thresholds will not behave correctly.

<task type="auto">
  <name>Task 2: Add durationMs timing to encoder-adapter.js flushQueue callback</name>
  <files>extension/sidebar/encoder-adapter.js</files>
  <action>
In `encoder-adapter.js`, modify the `flushQueue` function to measure encoding duration and pass it as a third argument to the `onBatchReady` callback.

Specifically, in the `flushQueue(onBatchReady)` function:
1. Before the `encodeMessages(batch)` call, record `const startTime = Date.now();`.
2. After `encodeMessages` resolves, compute `const durationMs = Date.now() - startTime;`.
3. Change the callback invocation from `onBatchReady(batch, embeddings)` to `onBatchReady(batch, embeddings, durationMs)`.

This is the minimal-impact change recommended in the research (Open Question 2). Existing callers that ignore the extra parameter are unaffected in JavaScript.

The timing measures actual inference time (inside flushQueue, wrapping encodeMessages), NOT queue wait time. This is correct per Pitfall 5 in research.
  </action>

  </verify>

flushQueue callback now receives (batch, embeddings, durationMs) where durationMs is the wall-clock milliseconds spent in encodeMessages() only (not queue wait time). Existing callers in sidebar.js that use (batch, embeddings) are unaffected.

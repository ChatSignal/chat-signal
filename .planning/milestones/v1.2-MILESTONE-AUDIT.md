## Partial Requirements Detail

### GPU-01 / GPU-02: LLM not routed through GPU scheduler

`llm-adapter.js` has zero imports — it does not use `scheduleGpuTask` from `gpu-scheduler.js`. WebLLM's GPU calls run through its own internal context, bypassing the promise-chain mutex entirely. Cross-subsystem serialization (encoder vs SLM) is architecturally absent.

**Why not critical:** WebLLM manages its own GPU context internally. The encoder frequently uses the WASM backend (no GPU contention). No runtime crashes or failures observed. The architectural guarantee is documented but unenforced — it's a design gap, not a functional gap.

**Recommendation:** Accept as tech debt. If GPU contention issues arise in practice, wrap LLM calls in `scheduleGpuTask('slm', 2, ...)`.

### SLM-04: LLM receives keyword buckets, not semantic buckets

`generateAISummary(result.buckets)` is called synchronously at sidebar.js:752 with WASM keyword-classified buckets. The cosine-routed semantic buckets are produced later in the async `scheduleEncode()` callback. The LLM never sees the semantic classification for the current cycle.

**Why not critical:** The LLM still receives meaningful bucket context (keyword-based classification is the original working system). The semantic upgrade only affects the UI cluster display, not the LLM prompt quality in a catastrophic way. A fix would require deferring `generateAISummary()` until after encoding completes.

**Recommendation:** Accept as tech debt. Fix in a future phase by moving `generateAISummary()` into the `scheduleEncode()` callback.
